What AETHER Is

AETHER is a deterministic reasoning engine.

It answers questions, solves problems, pursues goals, and assists users by processing language and applying structured logic — without a large language model anywhere in its core loop.

Every response AETHER produces is traceable. Every reasoning step follows an explicit rule. Every factual claim is retrieved from an authored knowledge corpus. There is no statistical generation, no probability-weighted word prediction, and no learned weights producing output that no one programmed directly.

This is not a limitation dressed up as a feature. It is a deliberate architectural decision with real consequences for what AETHER can guarantee and what it cannot.

What AETHER Is Not

Not
A large language model. It does not use transformer architecture. It does not have billions of parameters trained on internet text. It cannot generate plausible-sounding prose about topics it has never been explicitly given knowledge about. It will not pretend to know things it does not know.
Not
A chatbot wrapper. It does not sit in front of GPT, Claude, Gemini, or any other model. There is no API call to an external AI service in its reasoning path. The intelligence is in the architecture and the authored knowledge, not in a rented model.
Not
A general-purpose language model. It is an Authored General Intelligence system: its knowledge is what has been deliberately put into it, and its reasoning operates on that knowledge using explicit deterministic rules.
Not
Self-improving through training. It learns from corrections and new injected knowledge within defined mechanisms, but it does not retrain on interactions or update weights, because it has no weights to update.

Core Architecture

AETHER processes each turn through a sequential pipeline:

01
Understand
Input parsed for intent, frame, entities, session context, and ambiguity. Ambiguous inputs are held — a clarifying question is asked before reasoning begins.
02
Retrieve
Relevant knowledge fetched from the authored corpus using full-text search over aether_network. The lattice concept graph provides semantic enrichment. No generation.
03
Execute
Reasoning engine fires. Diagnostic hypotheses tracked with confidence. Deductive rules applied. Causal engine distinguishes correlation from causation. Actions dispatched.
04
Verify
Response checked against goal criteria and quality signals. Expansion modules: ADAPT, TRANSFER, CONSTRUCT, RECOMBINE, META_REFLECT.

Above this pipeline sit meta-control supervisors running in parallel: the Goal Monitor checks active goal expiry every turn, the Resource Allocator assigns a complexity tier and gates expensive stages, the WillEngine checks for standing intention violations before any work begins.

What AETHER Can Do

Factual retrieval with honest knowledge boundaries
Answers by retrieving from its authored corpus. When it has relevant content, it returns it with high precision. When it does not, it says so explicitly. It does not fabricate an answer to fill the silence. The system correctly says "I don't have knowledge on that" — this is the right behaviour, not a failure.
Multi-turn diagnostic reasoning
Maintains competing hypotheses across a conversation and updates confidence scores as evidence accumulates. Asks discriminating questions when two hypotheses are too close to separate. Falsifies hypotheses on contradicting evidence — distinct from mere outcompeting. Detects stuck threads and reframes rather than grinding. Deterministic: same inputs, same outputs.
Deductive reasoning with correctness guarantees
Syllogisms, logical consequence, set membership, rule application. "All employees over 40 hours get overtime. John worked 47 hours. Does he qualify?" — returns a deterministic, auditable answer. The reasoning step is explicit and inspectable. Rules injected in one turn are held in session state and applied in subsequent turns.
Causal reasoning: correlation vs. causation
Distinguishes temporal co-occurrence from causal mechanism. "Roosters crow before sunrise" triggers the correlational path — the engine flags the absence of mechanism and generates the counterfactual. "Smoking causes cancer because carcinogens damage DNA" triggers the causal path. This distinction is structural, not statistical.
Autonomous goal pursuit
Goals assigned with expiry conditions: turn limits, action limits, ISO 8601 deadlines, consecutive failure limits. Goal Monitor checks at every turn. The autonomous daemon pursues active goal steps between user turns — executing research queries and writing progress notes without requiring the user to keep the conversation active.
Real action execution
Action dispatcher executes shell commands, file system operations (read, write, delete, move, copy, search), HTTP requests, and API calls with real effects. Multi-step plans managed with dependency tracking and retry logic. Write operations in the daemon are gated behind explicit configuration — read-only is the default.
Learning from corrections and injections
Corrections are recorded in the persistent MistakeMemory store. When the same causal type is consistently miscorrected across sessions, LifelongLearningEngine registers a new problem type. New knowledge is indexed into FTS, fed to the lattice concept graph, and available immediately.
Session and organisational memory
Entities, topics, injected rules, stated intentions, affect trajectory, and situation context persist across turns. At session start, the engine pre-loads organisational vocabulary and problem type history from OrgModel — users in the same organisation benefit from what others have learned.

What AETHER Cannot Do

Generate knowledge it has not been given
AETHER cannot produce a coherent explanation of a topic not in its corpus. It will not confabulate. This is the sharpest difference from an LLM: an LLM will produce fluent and often plausible answers regardless of whether it actually knows the topic. That fluency masks uncertainty. AETHER's silence is honest.
Handle genuinely novel problem types without being shown them
The problem type registry starts with diagnostic, decision, optimisation, deductive, and causal query types. New types can be inferred from accumulated mistake patterns, but the inference mechanism is structural — pattern matching on causal type coverage — not generative. A genuinely novel class of problem will not be handled well until an appropriate framework is authored or inferred from sufficient evidence.
Understand language outside its structured parsing
The intent and frame classifiers were designed for a specific range of inputs. Highly creative, figurative, culturally specific, or domain-jargon-heavy language may be misclassified. The system has no statistical generalisation over the full distribution of language the way an LLM does.
Learn from raw unstructured exposure
AETHER does not learn by reading text the way a language model learns during pre-training. Injecting a raw document does populate the FTS index and feed the lattice learner, but the depth of understanding is limited by the quality and granularity of the source material. An LLM's latent representations generalise across its training distribution. AETHER's generalisation is limited to what has been explicitly authored.
Produce creative or generative content
AETHER does not write poems, stories, persuasive essays, or any open-ended generated content. Its prose renderer produces structured declarative sentences from reasoning outputs. Open-ended creative work requires the kind of statistical generation it deliberately lacks.
Guarantee correctness of authored knowledge
AETHER can guarantee that its reasoning is logically valid given its premises. It cannot guarantee that the premises are correct. If an article contains an error, the engine will correctly reason from that error and produce a wrong answer — but it will not have hallucinated. Wrong answers from bad data are identifiable and fixable; hallucinated answers from a language model are indistinguishable from correct ones without independent verification.

The Hallucination Guarantee

Hallucination in a large language model is not a bug. It is a structural consequence of the architecture: the model generates the statistically most probable continuation of the input, and sometimes the most probable continuation is factually wrong. The model has no mechanism to distinguish between "I know this" and "this sounds right."

Architectural guarantee

In AETHER, hallucination is architecturally impossible within the reasoning layer for a specific reason: the engine does not generate. Every factual claim in a response is retrieved from an article that was explicitly authored and reviewed. Every reasoning step applies an explicit rule. There is no probability distribution over tokens being sampled.

This guarantee has an important scope. It applies to the reasoning layer. If an article in the corpus contains incorrect information, the engine will correctly reason from that incorrect information and produce a wrong answer — but it will not have hallucinated.

The distinction matters: wrong answers from bad data are identifiable and fixable by correcting the corpus; hallucinated answers from a language model are indistinguishable from correct ones without independent verification.

Appropriate Use Cases

Well-suited for
✓
Correctness-consequential domains — health tech, EdTech, legal tech, financial services, compliance. These cannot tolerate confident wrong answers.
✓
Auditability requirements — every reasoning step is logged and inspectable. Regulators and auditors can trace how any conclusion was reached.
✓
Non-negotiable data privacy — AETHER runs entirely on-premise. User inputs never leave the deployment environment.
✓
Well-defined domain knowledge — the corpus can be authored with authority by domain experts. The more structured the domain, the better AETHER performs.
✓
Consistency requirements — the same question asked twice gets the same answer. No stochastic variation between runs.
Not suited for
✗
Open-domain conversation without a curated corpus — it will accurately report gaps rather than filling them fluently.
✗
Creative writing or content generation — it does not produce open-ended generated text.
✗
Tasks requiring generalisation from sparse data — LLMs generalise across training distributions; AETHER requires explicit knowledge coverage.
✗
Real-time information — unless the corpus is actively updated, AETHER's knowledge reflects the state of the corpus at injection time.

Deployment

AETHER deploys as a Docker stack: an API container running Flask with authentication middleware, a knowledge network server, an optional background daemon for autonomous goal pursuit, and an nginx frontend. All components run locally. No external AI API is called at runtime.

AETHER_ORG_ID
Enables organisational memory sharing across users in the same deployment.
AETHER_DAEMON_ENABLED
Starts the background worker for autonomous goal pursuit between user turns.
AETHER_DAEMON_WRITE
Enables file write operations in the daemon. Read-only is the default.
AETHER_API_KEY
Gates all API access. Injected by nginx — the browser never sees it.

New knowledge is injected via the /inject_domain endpoint or the inject_domain() Python method, which accepts a text or JSON corpus, indexes it into the FTS store, feeds it to the lattice concept graph, and optionally registers a YAML scaffold definition for structured task elicitation in that domain.

What "Authored General Intelligence" Means

The Authored General Intelligence designation describes the product of the breadth deployment: a system with authored knowledge across every major domain, providing useful and reliable answers across whatever a user brings to it.

The word "authored" is precise and intentional. Every piece of knowledge in the system was written by a person, reviewed, and deliberately included. The system's intelligence is the intelligence of its authors, expressed through a deterministic reasoning architecture.

This is a fundamentally different claim from the general intelligence claim made by or about large language models. Those systems acquire knowledge implicitly from statistical patterns in training data. AETHER acquires knowledge explicitly through deliberate authorship.

Neither approach is universally superior. Authored knowledge is more accurate, more auditable, and more correctable. Statistical learning generalises more broadly and requires no up-front authoring investment.

AETHER takes a clear position: in domains where correctness matters — health, law, education, finance — the auditability and correctness guarantees of authored knowledge outweigh the breadth advantage of statistical learning.

The appropriate comparison is not to a general-purpose LLM but to a domain expert working from a well-curated knowledge base, applying explicit reasoning rules, and knowing precisely the boundaries of their expertise.
